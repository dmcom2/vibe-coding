SECTION 01 · 철학

바이브 코딩의 4가지 핵심 철학

🎯
What > How (목적 중심)
"어떻게" 구현할지가 아닌 "무엇을" 만들지에만 집중합니다.
→ 구현의 디테일은 AI에게 맡기고, 개발자는 결과물의 가치에 집중합니다.
🗣️
자연어 인터페이스 (Natural Language)
함수 이름·문법 대신 일상 언어로 원하는 것을 설명합니다.
→ 코드는 AI와 인간 사이의 '중간 언어'가 되며, 자연어가 '최종 언어'가 됩니다.
🔄
반복 정교화 (Iterative Refinement)
한 번에 완벽할 필요 없습니다. 피드백을 통해 조금씩 다듬어 나갑니다.
→ '완벽한 설계'보다 '빠른 실행과 수정'이 더 효율적인 시대입니다.
🧭
개발자는 디렉터 (Director)
개발자 = 감독·의사결정자 / AI = 실행·구현 담당.
→ 기술적 구현 능력보다 '문제 정의 능력'과 '심미적 안목'이 더 중요해집니다.

SECTION 04 · Cline

클라인(Cline)의 4가지 핵심 특징

1. 내 손발이 되어주는 '자율형 에이전트(Agent)'
코드를 채팅창에 뱉어내고 끝나는 게 아닙니다. AI가 스스로 내 VS Code의 파일 탐색기를 뒤져서 코드를 읽고, 새 파일을 만들고, 기존 코드를 덮어쓰고, 심지어 터미널을 열어 서버를 실행하거나 오류를 테스트하는 등 '진짜 사람처럼' 행동합니다.
2. Plan(기획)과 Act(실행)의 분리
무작정 코드부터 짜지 않습니다. Plan 모드에서는 "이런 방식으로 작업할게"라고 먼저 기획서를 쓰고, 사용자가 오케이(Approve)하면 Act 모드로 넘어가 실제 코드를 수정합니다. 이 덕분에 대형 프로젝트에서도 방향성을 잃지 않습니다.
3. MCP (Model Context Protocol) 지원
클라인의 등 뒤에 무기를 달아주는 기능입니다. 깃허브(GitHub), 포스트맨(Postman), 브라우저 등 외부 도구를 연결해서, AI가 스스로 웹을 검색하거나 API를 테스트하게 만들 수 있는 엄청난 확장성을 가졌습니다.
4. 안전장치 (권한 승인)
AI가 내 컴퓨터를 마음대로 휘젓다 파일을 날려버리면 안 되겠죠? 파일을 수정하거나 터미널 명령어를 칠 때마다 반드시 사람(사용자)의 승인(Approve)을 받도록 설계되어 있어 안전하게 작업 과정을 통제할 수 있습니다.
SECTION 04 · Cline

클라인을 200% 활용하는 4대 비법

1. 두뇌(Model) 투자는 아끼지 마라
코딩 1티어 모델(Claude 3.5 Sonnet, Gemini 2.0 Flash Thinking 등)을 물려줘야 진가가 나옵니다. 멍청한 모델은 파일 경로도 못 찾습니다.
2. 고삐 쥐기: .clinerules 작성
프로젝트 최상단에 .clinerules 파일을 만들어 나만의 절대 규칙(예: "한국어만 써", "React 말고 순수 HTML만 써")을 적어두면 딴길로 새지 않습니다.
3. 타겟은 좁고 명확하게 (@ 활용)
추상적인 지시는 토큰 낭비입니다. @를 활용해 특정 파일이나 폴더를 지정하여 '단순 노동'을 시키는 것이 바이브 코딩의 정석입니다.
4. 꼬이면 '새 판 깔기(New Task)'
AI가 같은 에러를 반복하면 싸우지 마세요. 과감하게 + (New Task) 버튼을 눌러 과거 기억을 날리고 깨끗한 뇌 상태에서 다시 지시하세요.
SECTION 04 · .clinerules

클라인의 절대 헌법, .clinerules

왜 필요한가?
프로젝트 최상단에 이 파일 하나만 던져두면, AI가 헛발질하는 횟수가 10분의 1로 뚝 떨어집니다. AI의 뇌에 강제로 박아넣는 절대 헌법입니다.
1단계: 파일 생성
VS Code 루트 폴더에 마우스 우클릭 ➔ 새 파일(New File) ➔ .clinerules (맨 앞 마침표 필수!)
2단계: 템플릿 복사
마스터 템플릿을 복사해 붙여넣고 저장(Ctrl+S)하세요. 모든 에러를 원천 차단하는 범용 룰입니다.
💡
3단계: 기술 스택 추가
기본 헌법 아래에 프로젝트 특성(예: "Tailwind CSS 사용", "React 컴포넌트 분리")을 한두 줄 추가하면 완벽합니다.

마스터 템플릿 행동 가이드라인

# LLM 코딩 행동 가이드라인 이 가이드라인은 일반적인 LLM 코딩 실수를 줄이기 위한 행동 지침입니다. 프로젝트별 지침과 병합하여 사용하십시오. **기본 원칙:** 속도보다 신중함을 우선합니다. 매우 단순한 작업의 경우 판단하에 유연하게 적용하십시오. ## 1. 코딩 전 생각하기 **가정하지 마십시오. 혼란스러운 점을 숨기지 마십시오. 트레이드오프를 명확히 하십시오.** 구현 전 다음 사항을 준수하십시오. - 가정을 명시적으로 밝히십시오. 불확실한 경우 질문하십시오. - 여러 가지 해석이 가능한 경우 이를 제시하고, 임의로 하나를 선택하지 마십시오. - 더 단순한 접근 방식이 있다면 제안하십시오. 필요한 경우 요청에 대해 재고를 요청하십시오. - 불명확한 점이 있다면 중단하고, 무엇이 혼란스러운지 명시하여 질문하십시오. ## 2. 단순함 우선 **문제를 해결하는 최소한의 코드를 작성하십시오. 추측에 기반한 구현은 금지합니다.** - 요청하지 않은 기능은 추가하지 마십시오. - 단 한 번만 사용되는 코드에 추상화를 적용하지 마십시오. - 요청되지 않은 "유연성"이나 "설정 가능성"을 추가하지 마십시오. - 불가능한 시나리오에 대한 에러 핸들링을 하지 마십시오. - 200줄로 작성한 코드를 50줄로 줄일 수 있다면 다시 작성하십시오. "시니어 엔지니어가 보기에 너무 복잡한가?"라고 자문하고, 그렇다면 단순화하십시오. ## 3. 정밀한 수정 (Surgical Changes) **꼭 필요한 부분만 수정하십시오. 자신이 만든 문제만 정리하십시오.** 기존 코드를 수정할 때 다음 사항을 준수하십시오. - 인접한 코드, 주석 또는 포맷팅을 임의로 "개선"하지 마십시오. - 고장 나지 않은 부분을 리팩토링하지 마십시오. - 본인의 스타일과 다르더라도 기존 스타일을 유지하십시오. - 관련 없는 죽은 코드를 발견하면 언급하되, 요청 없이 삭제하지 마십시오. 변경 사항으로 인해 사용되지 않게 된 임포트, 변수, 함수는 제거하십시오. 단, 기존에 이미 있던 죽은 코드는 요청받지 않는 한 삭제하지 마십시오. 모든 변경된 라인은 사용자의 요청과 직접적으로 연결되어야 합니다. ## 4. 목표 중심 실행 **성공 기준을 정의하고 검증될 때까지 반복하십시오.** 작업을 검증 가능한 목표로 변환하십시오. - "유효성 검사 추가" → "유효하지 않은 입력에 대한 테스트 작성 및 통과" - "버그 수정" → "버그를 재현하는 테스트 작성 및 통과" - "X 리팩토링" → "변경 전후로 테스트가 통과하는지 확인" 다단계 작업의 경우 간결한 계획을 세우십시오. 1. [단계] → 검증: [체크 항목] 2. [단계] → 검증: [체크 항목] 3. [단계] → 검증: [체크 항목] 명확한 성공 기준이 있어야 독립적으로 루프를 돌며 완성도를 높일 수 있습니다. ## 5. 한국어 출력 시 마침표 사용 **한국어 문장은 콜론(:)이 아닌 마침표(.)로 끝내십시오.** 사용자가 한국어로 작성하면 응답도 한국어로 합니다. - 다음 줄에 리스트나 예시가 오더라도 문장을 콜론(:)으로 끝내지 마십시오. - 모든 한국어 문장 종결자는 `.`, `?`, `!` 중 하나여야 합니다. - 코드 내부, 키-값 쌍, 라벨에서는 콜론 사용이 가능하지만 문장 종결자로 사용하는 것은 금지합니다. ## 6. 한국어 파일 헤더 주석 **새로 생성하는 모든 소스 파일의 첫 줄에는 해당 파일의 역할을 설명하는 한 줄의 한국어 주석을 추가하십시오.** 파일 생성 시 다음과 같이 작성하십시오. - TypeScript/JavaScript: `// 사용자 인증 상태를 관리하는 Context Provider` - Python: `# KIS API 호출을 비동기로 래핑하는 클라이언트` - SQL: `-- 일별 집계 결과를 저장하는 머티리얼라이즈드 뷰` - 필수 지시어(`'use client'`, `'use server'`, shebang 등) 바로 아래에 배치하십시오. - 설정 파일(`*.config.ts`, `package.json` 등)은 제외합니다. ## 7. 계획 + 체크리스트 + 컨텍스트 노트 **사소하지 않은 모든 작업 전에 다음 세 가지 결과물을 생성하십시오. 이것 없이 코딩을 시작하지 마십시오.** - **계획 (Plan)**: 무엇을 왜 만드는지에 대한 설명. - **체크리스트 (checklist.md)**: 체크박스 형태의 구체적인 작업 목록. 진행 상황에 따라 체크하십시오. - **컨텍스트 노트 (context-notes.md)**: 작업 중 내린 결정과 그 이유를 지속적으로 기록하십시오. 사용자가 계획만 주고 코딩을 시작하라고 요청한다면, 먼저 체크리스트와 컨텍스트 노트를 생성해야 하는지 확인하십시오. ## 8. 완료 전 테스트 실행 **코드를 수정했다면 완료라고 말하기 전에 반드시 테스트를 실행하십시오.** - `npm test`, `pytest`, `cargo test` 등 프로젝트에서 사용하는 테스트 도구를 실행하십시오. - 테스트가 통과하면 결과를 보고하고, 실패하면 수정 후 다시 실행하십시오. - 테스트 설정이 없다면 최소한 프로젝트가 빌드/컴파일되는지 확인하십시오. - 사용자가 "끝", "완료"라고 말하기 전에 선제적으로 테스트를 수행하십시오. ## 9. 시맨틱 커밋 (Semantic Commits) **하나의 논리적 변경이 완료되었을 때 커밋하십시오. 사용자가 요청할 때까지 기다리지 마십시오.** - "이 커밋을 한 문장으로 설명할 수 있는가?"라고 자문하십시오. 가능하다면 커밋하십시오. - 좋은 예: "auth 미들웨어 추가". 나쁜 예: "auth 추가하고 UI도 고치고 버그도 수정" (3개로 분리하십시오). - 너무 많은 무관한 수정 사항을 한꺼번에 커밋하여 개별 롤백 능력을 상실하지 마십시오. ## 10. 에러 읽기, 추측 금지 **실제 에러 로그 라인을 읽으십시오. 기억에 의존해 패턴 매칭을 하지 마십시오.** 실패가 발생했을 때: - 전체 에러 메시지와 스택 트레이스를 읽으십시오. - 추측하지 말고 실제 로그 출력을 확인하십시오. - 원인을 확인하기 전에 "일반적인 해결책"을 적용하지 마십시오. - 불분명한 경우 로그를 추가하여 상태를 확인한 후 수정하십시오. ## 11. 언어 설정 **모든 설명과 응답은 항상 한글로 작성하십시오.** - 코드 주석, 커밋 메시지, 사용자 응답, 기술 설명 등 모든 소통 과정에서 한글 사용을 원칙으로 합니다.

SECTION 05 · 시작하기

GitHub & Cline 완전 정복

1. GitHub 가입 및 저장소 생성
github.com 가입 후, 'New repository'에서 영문 이름으로 저장소를 만듭니다.
2. 토큰 발급 (필수 권한)
Settings ➔ Developer settings ➔ Personal access tokens ➔ Tokens(classic) ➔ 'Generate new token'.
[필수 체크] repo (저장소 접근), workflow (자동화 워크플로우)
3. 작업 공간 및 연결
# 로컬 폴더 생성 및 초기화 mkdir my-vibe-project && cd my-vibe-project git init # 원격 저장소 연결 및 푸시 (비밀번호에 토큰 입력) git remote add origin https://github.com/사용자명/저장소명.git git add . && git commit -m "Initial commit" git push -u origin master
4. commit과 push, 무슨 뜻인가요?
📸 git commit — "저장"
지금까지 수정한 파일들을 내 컴퓨터(로컬)에만 하나의 버전(스냅샷)으로 기록합니다. 아직 GitHub에는 올라가지 않은 상태입니다.
git add .
git commit -m "로그인 기능 추가"
☁️ git push — "업로드"
로컬에 쌓인 commit들을 GitHub 원격 저장소로 전송합니다. push를 해야 비로소 팀원이나 다른 PC에서 내 변경사항을 볼 수 있습니다.
git push
비유: commit은 게임의 "저장(세이브포인트)", push는 그 저장 파일을 "클라우드에 업로드"하는 것과 같습니다. 작업 단위가 끝날 때마다 자주 commit하고, 어느 정도 정리됐을 때 push하세요. (흐름: add → commit → push)
5. 연결 확인
터미널에서 git remote -v 명령어를 입력하여 원격 저장소 주소가 올바르게 표시되는지 확인합니다.

🌐 맥락 (Context)
어떤 상황에서, 왜 이 도구가 필요한지 배경 정보를 제공합니다.
"매주 20명 정도의 팀원이 참석하는 주간 회의에서 사용할 거야"
3
👤 사용자 (User)

"IT 지식이 없는 일반 직원들도 쉽게 쓸 수 있어야 해"
4
📋 기능 (Feature)
필요한 핵심 기능을 구체적인 동작 단위로 나열합니다.
"이름 추가/삭제, 원클릭 출석 체크, 브라우저 저장(localStorage)"

🎨 디자인 (Style)

"애플 스타일의 깔끔한 화이트 톤, 포인트 컬러는 딥 퍼플"
6
🔧 제약 (Constraint)
기술적 제한 사항이나 반드시 지켜야 할 규칙을 명시합니다.
"외부 라이브러리 없이 HTML 단일 파일로 구현, 한국어 주석 필수"

반드시 알아야 할 4가지 리스크

보안 위험
  • AI는 기능 구현 우선, 보안은 후순위
  • XSS, SQL 인젝션 취약점 자동 방어 ✗
  • API 키를 코드에 직접 노출할 수 있음
→
민감 데이터 처리 시 '보안 취약점 점검'을 명시적으로 요청하고, API 키는 .env 파일로 분리하세요.
코드 품질
  • 불필요한 함수·중복 코드 발생
  • 일관성 없는 스타일, 주석 부족
  • 성능 최적화 없이 동작만 구현
→
기능 구현 후 '코드 리팩토링 및 최적화'를 요청하여 중복을 제거하고 가독성을 높이세요.
AI 의존성
  • 코드 내용을 이해 못 한 채 배포 위험
  • 에러 원인 파악이 점점 어려워짐
  • 점진적 코딩 실력 저하 가능
→
AI가 작성한 핵심 로직의 흐름을 반드시 검토하고, 중요한 부분은 직접 주석을 달아 기록하세요.
유지보수 어려움
  • 의사결정 추적 부재 (왜 이렇게 했는지 모름)
  • 팀 협업 시 코드 이해 장벽
  • 대규모 프로젝트에 단독 사용 부적합
→
프로젝트 구조와 사용법을 담은 README.md 작성을 요청하여 유지보수 가능성을 확보하세요.
SECTION 08 · 주의사항

AI 환각(Hallucination) 대처법

😵
환각 현상이란?
AI가 매우 자신감 있게 틀린 정보나 존재하지 않는 함수/라이브러리를 마치 사실인 것처럼 제공하는 현상입니다.
실행 기반 검증 (Run & Verify)
AI의 설명을 믿지 말고 '실행 결과'를 믿으세요. 코드를 즉시 실행해보고 의도대로 동작하는지 눈으로 확인하는 것이 가장 정확합니다.
구체적 오류 지적
"그냥 안 돼"라고 하기보다 "X 함수에서 Y 에러가 발생해"라고 정확한 지점을 알려주세요. AI가 오류를 수정할 확률이 비약적으로 높아집니다.
맥락 재설정 (Reset Context)
AI가 같은 실수를 반복하며 루프에 빠졌다면, 현재 상태를 요약해 다시 알려주거나 새 세션을 시작해 불필요한 이전 기억을 지우세요.
레퍼런스 제공
공식 문서의 일부나 원하는 정확한 예시 코드를 프롬프트에 포함하세요. AI가 참고할 '정답지'를 주면 환각이 현저히 줄어듭니다.
💡
바이브 코딩의 핵심은 AI를 맹신하는 것이 아니라, AI의 결과물을 끊임없이 의심하고 검증하는 '디렉터'의 역량에 있습니다.
