1부 · DB가 왜 필요한가

잘 쓰던 엑셀·구글시트가 힘들어지는 순간

동시 편집 충돌
재고 시트를 영업팀과 창고팀이 동시에 열어 각자 수량을 고치면, 나중에 저장한 사람 것만 남고 먼저 고친 내용은 흔적도 없이 사라집니다. DB는 "누가 언제 뭘 바꿨는지"를 기록으로 남기고, 동시에 같은 값을 덮어쓰지 못하도록 원천적으로 막습니다.
데이터 중복·불일치
담당자가 바뀌면 "거래처 시트"와 "발주 시트"를 각각 찾아가 고쳐야 하는데, 하나라도 빠뜨리면 그 순간부터 두 표는 서로 다른 사실을 말하게 됩니다. DB는 정보를 한 곳에만 저장하고 나머지 표는 그 값을 "연결"해서 쓰기 때문에 이런 불일치가 구조적으로 생기지 않습니다 — 이 원칙이 바로 2부에서 배울 "관계"의 핵심 아이디어입니다.
관계 조회의 어려움
"이 고객이 지난달에 뭘 샀지?"에 답하려고 VLOOKUP·XLOOKUP을 3~4겹 중첩해서 쓰는 상황 — 수식 하나가 깨지면 전체 결과가 조용히 틀어집니다. DB는 이런 질문을 SQL 한 줄로, 게다가 원본 수식이 깨질 걱정 없이 끝냅니다.
엑셀이 나쁘다는 뜻이 아닙니다. 1~4편에서 배운 VBA·GAS 자동화는 여전히 훌륭한 도구입니다. 다만 데이터 양이 늘고, 여러 사람이 같이 쓰고, 표 사이의 관계가 복잡해질수록 DB가 훨씬 안전하고 빠릅니다. 체감으로는 한 시트가 몇만 행을 넘거나 열 때마다 버벅이기 시작하면 이미 넘어갈 시점을 지난 신호입니다.
1부 · DB가 왜 필요한가

언제까지 시트로 버티고 언제 DB로 넘어가야 하나

아직은 시트로 충분한 경우
· 나 혼자, 또는 소수만 다루는 데이터
· 몇백~몇천 행 이내, 표 하나로 끝나는 단순 구조
· 매번 통째로 열어보고 눈으로 확인해도 괜찮은 규모
· "지난달 것도 궁금하네" 정도로, 과거 데이터 조회가 드문 경우
DB로 넘어가야 할 신호
· 여러 명이 동시에 같은 데이터를 읽고 쓴다
· "고객-주문", "회원-예약"처럼 표 2개 이상이 서로 연결된다
· 데이터가 수만 행을 넘고, 계속 쌓인다
· 실수로 지워지면 안 되는 중요한 데이터라 백업·복원이 필요하다
· "이 조건에 맞는 것만" 같은 검색·필터링을 자주, 빠르게 해야 한다
판단이 애매하면 "지금 쓰는 시트, DB로 옮기는 게 나을까?"라고 AI에게 현재 상황을 그대로 설명하고 물어보는 것도 방법입니다. 다만 오늘 실습이 끝나면 이 판단을 "느낌"이 아니라 직접 만들어본 경험으로 할 수 있게 됩니다 — DB가 뭔지 모르는 상태에서 묻는 것과, 한 번이라도 만들어본 뒤 묻는 것은 질문의 질이 완전히 다릅니다.

2부 · 핵심 개념

테이블 = 시트, 행 = 한 줄, 기본키 = 사번

테이블(Table)
엑셀의 시트 하나와 같습니다. "회원" 테이블, "주문" 테이블처럼 성격이 같은 데이터를 모아둔 표입니다.
행(Row) = 레코드
엑셀의 가로 한 줄입니다. "회원 테이블"의 한 행 = 회원 한 명의 정보 전체.
열(Column) = 필드
엑셀의 세로 한 칸입니다. 이름, 연락처, 가입일처럼 "어떤 종류의 정보인지"를 정의합니다. 값이 아직 없을 수도 있는데, 이 "빈 상태"를 DB에서는 NULL이라 부릅니다 — 숫자 0이나 빈 문자열과는 다른, "아예 모른다"는 뜻입니다.
기본키(Primary Key)
그 행을 절대 안 겹치게 구분해주는 값. 직원의 사번, 학생의 학번 같은 것. 만약 "김철수"란 이름의 회원이 두 명이면, "김철수 등급을 VIP로 바꿔줘"라는 요청이 둘 중 누구에게 적용될지 알 수 없게 됩니다 — 그래서 이름 대신, 절대 안 겹치는 회원ID 같은 값을 기본키로 씁니다.
제약조건(Constraint)
"이름은 절대 비어있으면 안 돼(NOT NULL)", "연락처는 서로 겹치면 안 돼(UNIQUE)"처럼 DB에게 지켜야 할 규칙을 미리 정해두는 것. 사람이 매번 확인할 필요 없이 DB가 자동으로 규칙을 어긴 입력을 막아줍니다. 실습 1에서 "이름은 필수 입력"이라고 요청하는 게 바로 이 제약조건입니다.
// "회원" 테이블 예시

회원ID(PK) | 이름 | 연락처 | 가입일 | 등급
─────────────────────────────
1 | 김철수 | 010-1234 | 2026-01-05 | 일반
2 | 이영희 | 010-5678 | 2026-02-14 | VIP
3 | 박민수 | NULL | 2026-03-02 | 일반

// 회원ID가 기본키 → 무조건 유일한 값
// 박민수는 연락처를 아직 안 알려줘서 NULL
// (이름 열엔 NOT NULL이 걸려있어 비워둘 수 없음)
2부 · 핵심 개념

테이블끼리 연결(관계)되는 방식

"고객" 표와 "주문" 표, 회원ID로 연결됩니다
고객 테이블
회원ID·이름·연락처
회원ID로 연결
주문 테이블
주문ID·회원ID·상품·수량
고객 한 명이 주문을 여러 번 할 수 있으니 1(고객) : N(주문) 관계입니다. "김철수가 뭘 샀지?"는 두 표를 회원ID로 이어 붙이면 바로 답이 나옵니다. 이렇게 다른 테이블의 기본키를 가져와 연결에 쓰는 값을 외래키(Foreign Key, FK)라고 부릅니다 — 주문 테이블 안의 회원ID가 바로 외래키입니다.

"이거 사실 엑셀에서 VLOOKUP·XLOOKUP으로 매일 하던 일입니다. DB의 관계(JOIN)는 그 VLOOKUP을 훨씬 빠르고 안전하게 자동으로 처리해주는 것뿐입니다."

참고로 관계에는 1:1, 1:N, N:M 세 종류가 있는데 실무에서 가장 흔한 건 오늘 본 1:N입니다. 오늘 실습에서 직접 다룰 표는 "회원" 테이블 하나뿐이니, 관계는 개념만 이해하고 넘어가도 충분합니다 — 표 하나만 잘 다뤄도 오늘의 목표는 달성입니다.

SQL, 외울 필요 없습니다 — 얼굴만 트세요

SELECT= Read
SELECT = "보여줘" — 데이터 조회. 조건을 걸 땐 뒤에 WHERE("~인 것만"), 순서를 정할 땐 ORDER BY("~순으로 정렬")를 붙입니다.
INSERT= Create
INSERT = "넣어줘" — 새 데이터 추가
UPDATE= Update
UPDATE = "고쳐줘" — 기존 데이터 수정. WHERE를 빠뜨리면 전체 행이 다 바뀌니 항상 짝지어 씁니다.
DELETE= Delete
DELETE = "지워줘" — 데이터 삭제. 역시 WHERE 없이 쓰면 테이블 전체가 통째로 사라집니다.
-- "VIP 등급 회원을, 최근 가입한 순으로 보여줘"
-- 를 SQL로 쓰면

SELECT * FROM 회원
WHERE 등급 = 'VIP'
ORDER BY 가입일 DESC;

-- 이 문장을 직접 외워 쓸 필요가 없습니다
-- "VIP 등급만, 최근 가입 순으로 조회하는
-- SQL 짜줘"라고 AI에게 말하면 됩니다
오늘 실습의 원칙: SQL 문법은 AI가 씁니다. 여러분은 "무엇을, 왜" 필요한지 정확히 말하고, AI가 짠 SQL을 pgAdmin에 붙여넣고 실행 버튼만 누르면 됩니다. 1편부터 계속 써온 패턴과 완전히 같습니다. 참고로 이 4개 명령(만들고·읽고·고치고·지우고)을 묶어서 CRUD(Create·Read·Update·Delete)라고 부르는데, 3부 실습에서 이 이름 그대로 다시 만납니다.

DB 교육에서 제일 실무적으로 중요한 것

"열심히 만든 테이블을 실수로 통째로 지워봤다가, 백업 파일 하나로 되살려보는 경험 — 이걸 한 번이라도 직접 해본 사람과 안 해본 사람은 실무에서 다릅니다."

백업(Backup)
데이터베이스 우클릭 → Backup... → 파일 이름 지정 → Backup 버튼. DB 전체가 .backup 파일 하나로 저장됩니다.
복원(Restore)
새 빈 데이터베이스 생성 → 우클릭 → Restore... → 아까 만든 백업 파일 선택. 지워졌던 데이터가 그대로 돌아옵니다.
이번 실습에서는 일부러 DROP TABLE로 테이블을 진짜로 삭제해본 뒤, 방금 만든 백업으로 되살리는 과정까지 체험합니다. "지워도 괜찮다"는 안전감을 몸으로 익히는 게 목표입니다.

DB를 쓰기 시작하면 지켜야 할 최소한의 위생

비밀번호 노출 금지
postgres 관리자 비밀번호는 카카오톡·이메일로 절대 그대로 보내지 않습니다. 코드 안에 그대로 적어놓고 깃허브 같은 공개 저장소에 올리는 것도 같은 실수입니다.
외부에 포트 열지 않기

실습 1 · AI에게 시켜 회원 테이블 만들기
난이도 ★☆☆ · 약 10분 · 3부 설치·실습
상황
S12까지 따라왔다면 pgAdmin에서 practice_db를 만들고 Query Tool을 열어둔 상태입니다. 이제 그 안에 "회원 관리"용 테이블을 하나 만듭니다. SQL 문법은 몰라도 됩니다 — AI에게 요구사항만 정확히 전달합니다.
1단계 — AI에게 CREATE TABLE 시키기
PostgreSQL에서 회원 관리용 테이블을 만들려고 해. 아래 조건에 맞는 CREATE TABLE 문을 짜줘. [테이블 이름] members [컬럼 구성] - 회원ID: 자동으로 1씩 증가하는 기본키 - 이름: 필수 입력 (비어있으면 안 됨) - 연락처: 문자열 - 가입일: 날짜, 별도로 입력 안 하면 오늘 날짜가 자동으로 들어가게 - 등급: 문자열, 별도로 입력 안 하면 '일반'이 자동으로 들어가게 [조건] - PostgreSQL 최신 문법 기준으로 작성 (자동증가는 SERIAL이나 GENERATED ALWAYS AS IDENTITY 중 요즘 권장되는 방식으로) - 컬럼명은 전부 소문자로 적어줘 (예: 회원id, 이름, 연락처, 가입일, 등급) — 나중에 파이썬에서 똑같은 이름으로 다시 불러올 거라 대소문자를 지금 통일해두는 게 중요해 - 완성 후 "pgAdmin Query Tool에 붙여넣고 실행하는 방법"도 한 문단으로 설명해줘
2단계 — Query Tool에서 실행
AI가 준 SQL 전체를 복사해 Query Tool 빈 화면에 붙여넣기
키보드 F5 또는 상단 ▶(Execute) 버튼 클릭
하단에 "Query returned successfully" 메시지가 뜨면 성공
왼쪽 트리의 Tables를 우클릭 → Refresh → members 테이블이 보이는지 확인
에러가 났다면
당황하지 말고 화면 하단에 빨간 글씨로 뜬 에러 메시지 전체를 그대로 복사해 AI에게 붙여넣고 "이 에러 고쳐줘"라고 요청하세요. 해석하려 하지 말고 그대로 전달하는 것이 가장 빠릅니다.
체크포인트
Tables 목록에 members 테이블이 보이는가?
members 우클릭 → Properties → Columns 탭에서 5개 컬럼이 모두 있는가?
실습 2 · AI로 CRUD 전 과정 해보기
난이도 ★★☆ · 약 15분 · 3부 설치·실습
상황
실습 1에서 만든 members 테이블은 아직 비어있습니다. 이번엔 데이터를 넣고(Create), 조회하고(Read), 고치고(Update), 지우는(Delete) 4단계를 순서대로 AI에게 시켜서 진행합니다. 매 단계마다 Query Tool에 붙여넣고 F5로 실행하세요.
Create — 샘플 회원 데이터 넣기
members 테이블(회원ID·이름·연락처·가입일·등급)에 샘플 회원 10명을 넣는 INSERT문을 짜줘. - 이름은 실제로 있을 법한 한글 이름으로 - 연락처는 010-XXXX-XXXX 형식으로 - 등급은 '일반' 7명, 'VIP' 3명 정도로 섞어서 - 회원ID와 가입일은 자동으로 채워지니 값을 직접 넣지 말아줘
Read — 조건별로 조회하기
members 테이블에서 다음 3가지를 조회하는 SELECT문을 각각 짜줘. 1) 등급이 VIP인 회원만 보여줘 2) 전체 회원을 가입일이 최근인 순서로 정렬해서 보여줘 3) 전체 회원 수를 숫자 하나로 세어줘
Update — 잘못된 값 고치기
(방금 조회 결과에서 회원ID 하나를 골라서) 회원ID가 [숫자]인 회원의 등급을 'VIP'로 변경하는 UPDATE문을 짜줘. WHERE 조건을 반드시 회원ID로 걸어줘 — 조건을 빼먹으면 전체 데이터가 다 바뀌니까 조심해야 한다고 경고 문구도 같이 알려줘
Delete — 필요없는 데이터 지우기
(다시 회원ID 하나를 골라서) 회원ID가 [숫자]인 회원 한 명만 삭제하는 DELETE문을 짜줘. 역시 WHERE 조건을 회원ID로 정확히 걸어서, 다른 회원까지 같이 지워지지 않게 해줘
체크포인트
UPDATE·DELETE를 실행하기 전 WHERE 조건이 있는지 AI 답변에서 반드시 확인했는가? (WHERE 없이 실행하면 테이블 전체가 바뀌거나 몽땅 지워집니다)
View/Edit Data → All Rows로 최종 데이터가 예상대로 바뀌었는지 눈으로 확인했는가?
응용 미션
"이름에 '김'이 들어간 회원만 보여줘", "가입일이 이번 달인 회원 수를 세어줘"처럼 나만의 조건으로 SELECT를 3개 더 만들어보세요. 조건을 바꿔가며 요청하는 것만으로 SQL의 감이 빠르게 잡힙니다.
실습 3 · 백업하고, 지우고, 복원하기
난이도 ★★☆ · 약 10분 · 3부 설치·실습
상황
지금 practice_db에는 실습 2에서 넣은 회원 데이터가 들어있습니다. 이 데이터를 백업 파일로 저장한 뒤, 일부러 테이블을 통째로 지워보고, 백업으로 되살립니다. "실수해도 괜찮다"는 감각을 손으로 익히는 게 목적입니다.
STEP 1 — 백업하기
왼쪽 트리에서 practice_db 우클릭 → Backup...
Filename에 저장 경로 지정 (예: 바탕화면에 practice_db_backup)
Format은 기본값(Custom) 그대로 두고 Backup 버튼 클릭
"Backup completed successfully" 메시지가 뜨면 성공 — 파일이 실제로 생겼는지 탐색기에서 확인
STEP 2 — 일부러 지워보기
이제 정말로 테이블을 삭제합니다. 이 명령은 되돌릴 수 없으니, 방금 백업을 마쳤는지 한 번 더 확인한 뒤 진행하세요.
DROP TABLE members;
위 문장을 Query Tool에 붙여넣고 F5 실행 → members 테이블이 목록에서 사라진 것을 확인
STEP 3 — 복원으로 되살리기
practice_db 우클릭 → Restore...
Filename에서 STEP 1에서 만든 백업 파일 선택
Restore 버튼 클릭 → 완료 메시지 확인
Tables 새로고침 → members 테이블과 데이터가 그대로 돌아왔는지 확인
체크포인트
DROP TABLE로 지운 뒤 정말로 사라진 것을 눈으로 봤는가?
복원 후 View/Edit Data로 데이터 개수가 실습 2 때와 같은지 확인했는가?
응용 미션
실무에서는 "매일 자정에 자동으로 백업"처럼 주기적인 백업이 중요합니다. 여유가 되면 AI에게 "PostgreSQL을 매일 자동으로 백업하는 방법을 알려줘"라고 물어보고 답변만 읽어보세요 — 오늘 당장 설정하지 않아도, 이런 방법이 있다는 것만 알아둬도 충분합니다.
실습 4 · 파이썬으로 내 DB 접속하기
난이도 ★★★ · 약 15분 · 4부 파이썬 연결 (선택)
상황
지금까지는 pgAdmin이라는 GUI로만 DB를 다뤘습니다. 이번엔 진짜 프로그램(파이썬)이 내 DB에 직접 접속해서, 조회는 물론 추가·수정·삭제까지 되는 미니 "회원 관리 프로그램"을 완성합니다. 파이썬 문법은 몰라도 됩니다 — AI가 짜줍니다.
STEP 1 — 라이브러리 설치
VS Code에서 터미널 열기(상단 메뉴 터미널 → 새 터미널)를 하고 아래 명령어를 입력합니다. PostgreSQL과 파이썬을 연결해주는 도구를 설치하는 명령입니다.
pip install psycopg2-binary
"Successfully installed psycopg2-binary" 메시지가 뜨면 성공
STEP 2 — 먼저 조회부터 (첫 연결 성공시키기)
한 번에 다 만들려 하지 말고, 먼저 조회 하나만 되는 스크립트로 접속이 되는지부터 확인합니다.
파이썬으로 로컬 PostgreSQL에 접속해서 members 테이블의 모든 데이터를 가져와 출력하는 스크립트를 짜줘. [연결 정보] - 호스트: localhost - 포트: 5432 - 데이터베이스: practice_db - 사용자: postgres - 비밀번호: 설치할 때 만든 비밀번호 (변수로 분리해줘) [조건] - psycopg2 라이브러리 사용 - 비밀번호 같은 민감정보는 코드 맨 위에 변수로 분리해서 눈에 잘 띄게 해줘 - 총 회원 수와 각 행의 데이터를 보기 좋게 출력해줘 - 마지막엔 연결을 반드시 닫아줘
db_test.py로 저장 → 비밀번호 변수에 실제 비밀번호 입력 → 터미널에서 python db_test.py 실행 → 회원 데이터가 출력되면 성공
STEP 3 — 추가·수정·삭제까지 되는 프로그램으로 확장
조회가 됐다면 이제 진짜 "프로그램"답게 확장합니다. 실행하면 메뉴가 뜨고, 번호만 입력하면 조회·추가·수정·삭제를 전부 할 수 있게 만듭니다.
방금 만든 조회 스크립트를 확장해서, 실행하면 메뉴가 뜨고 번호를 입력해 아래 기능을 전부 쓸 수 있는 회원 관리 프로그램으로 만들어줘. [메뉴 구성] 1) 전체 회원 조회 2) 회원 추가 (이름·연락처·등급을 입력받아서) 3) 회원 수정 (회원ID를 입력받아 등급을 바꾸기) 4) 회원 삭제 (회원ID를 입력받아 삭제, 삭제 전 "정말 삭제할까요? (y/n)"로 한 번 더 확인) 0) 종료 [조건] - input()으로 값을 입력받아줘 - 추가·수정·삭제는 값을 문자열로 직접 이어붙이지 말고 %s 플레이스홀더 방식으로 넣어줘 (SQL 인젝션 방지) - 수정·삭제할 때 입력한 회원ID가 존재하지 않으면 "해당 회원ID가 없습니다"라고 알려줘 - 각 작업이 끝나면 다시 메뉴로 돌아가는 반복 구조로 만들어줘 (0번을 눌러야 종료)
STEP 4 — 저장하고 4가지 기능 모두 써보기
AI가 준 전체 코드로 db_test.py를 덮어쓰고 저장(Ctrl+S) — 비밀번호 변수는 다시 확인
터미널에서 python db_test.py 실행 → 메뉴가 뜨는지 확인
2번(추가)로 새 회원 한 명을 넣고, 1번(조회)로 실제로 늘어났는지 확인
3번(수정)으로 방금 넣은 회원의 등급을 바꾸고, 다시 1번으로 확인
4번(삭제)로 그 회원을 지우고, 1번으로 사라졌는지 확인 → pgAdmin의 View/Edit Data에서도 같은 결과가 보이면 완성
참고 · 검증된 예시 코드 (막히면 그대로 쓰세요)
STEP 1에서 만든 members 테이블의 컬럼명이 회원id·이름·연락처·가입일·등급(전부 소문자)이라고 가정한 코드입니다. pgAdmin에서 members 우클릭 → Properties → Columns 탭에서 실제 컬럼명과 다르면 그 부분만 맞게 바꿔주세요.
import psycopg2 # 접속 정보 (민감정보이므로 실습 후 비밀번호는 지우거나 가려두세요) DB_HOST = "localhost" DB_PORT = "5432" DB_NAME = "practice_db" DB_USER = "postgres" DB_PASSWORD = "여기에_설치할_때_설정한_비밀번호" def get_connection(): return psycopg2.connect( host=DB_HOST, port=DB_PORT, dbname=DB_NAME, user=DB_USER, password=DB_PASSWORD ) def list_members(): conn = get_connection() cur = conn.cursor() cur.execute("SELECT 회원id, 이름, 연락처, 가입일, 등급 FROM members ORDER BY 회원id;") rows = cur.fetchall() print(f"\n총 {len(rows)}명의 회원이 있습니다.") for row in rows: print(f" {row[0]} | {row[1]} | {row[2]} | {row[3]} | {row[4]}") cur.close() conn.close() def add_member(): 이름 = input("이름: ") 연락처 = input("연락처: ") 등급 = input("등급 (일반/VIP, 엔터만 누르면 일반): ") or "일반" conn = get_connection() cur = conn.cursor() cur.execute( "INSERT INTO members (이름, 연락처, 등급) VALUES (%s, %s, %s);", (이름, 연락처, 등급) ) conn.commit() cur.close() conn.close() print("추가되었습니다.") def update_member(): 회원id = input("등급을 바꿀 회원ID: ") 새등급 = input("새 등급 (일반/VIP): ") conn = get_connection() cur = conn.cursor() cur.execute( "UPDATE members SET 등급 = %s WHERE 회원id = %s;", (새등급, 회원id) ) conn.commit() if cur.rowcount == 0: print("해당 회원ID가 없습니다.") else: print(f"{cur.rowcount}명의 등급이 변경되었습니다.") cur.close() conn.close() def delete_member(): 회원id = input("삭제할 회원ID: ") 확인 = input(f"정말 회원ID {회원id}를 삭제할까요? (y/n): ") if 확인.lower() != "y": print("취소되었습니다.") return conn = get_connection() cur = conn.cursor() cur.execute("DELETE FROM members WHERE 회원id = %s;", (회원id,)) conn.commit() if cur.rowcount == 0: print("해당 회원ID가 없습니다.") else: print(f"{cur.rowcount}명이 삭제되었습니다.") cur.close() conn.close() def main(): while True: print("\n===== 회원 관리 프로그램 =====") print("1. 전체 조회 2. 추가 3. 수정(등급) 4. 삭제 0. 종료") 선택 = input("선택: ") if 선택 == "1": list_members() elif 선택 == "2": add_member() elif 선택 == "3": update_member() elif 선택 == "4": delete_member() elif 선택 == "0": print("프로그램을 종료합니다.") break else: print("잘못된 선택입니다.") if __name__ == "__main__": main()

column "회원id" does not exist → 실습 1에서 테이블을 만들 때 컬럼명이 예시와 다르게 생성된 경우. pgAdmin에서 실제 컬럼명을 확인해 코드의 컬럼명을 맞춰주세요

원리 한 줄
재설치할 필요 없습니다. 설정 파일 하나를 잠깐 "비밀번호 안 물어보기" 모드로 바꿔서 들어간 뒤, 새 비밀번호로 바꾸고 다시 원래대로 되돌리는 것이 전부입니다. 데이터는 전혀 건드리지 않습니다.
복구 순서
1
pg_hba.conf 파일 열기
보통 C:\Program Files\PostgreSQL\17\data\pg_hba.conf에 있습니다 (17은 설치한 버전 번호로 바꿔서). 메모장으로 열면 됩니다.
2
인증 방식을 잠깐 trust로 바꾸기
127.0.0.1/32, ::1/128, local로 시작하는 줄들의 맨 끝 컬럼(scram-sha-256 또는 md5)을 trust로 바꿔서 저장합니다. trust = "비밀번호 안 물어보고 그냥 들여보내줘"라는 뜻입니다.
3
PostgreSQL 서비스 재시작
윈도우 검색에서 services.msc 실행 → postgresql-x64-17 같은 이름 찾기 → 우클릭 → 다시 시작. 설정 파일을 바꿔도 서비스를 재시작해야 반영됩니다.
4
비밀번호 없이 접속해서 새 비밀번호 설정
pgAdmin에서 비밀번호 입력창에 아무거나 입력하거나 비워도 접속됩니다. 접속 후 Query Tool에서 아래 명령 한 줄을 실행합니다.
5
★ 반드시★ trust를 원래대로 되돌리기
2번에서 바꾼 부분을 scram-sha-256(원래 값)으로 되돌리고 저장 → 서비스 다시 재시작. 이 단계를 빼먹으면 누구나 비밀번호 없이 들어올 수 있는 상태로 남습니다.
4단계에서 쓰는 명령어
ALTER USER postgres WITH PASSWORD '새로_정할_비밀번호';
trust 모드인 동안은 누구나 비밀번호 없이 DB에 들어올 수 있는 상태입니다. 4번(비밀번호 재설정)을 마쳤으면 곧바로 5번(원상복구)을 진행하세요 — trust 상태로 오래 켜두지 않는 것이 중요합니다.
