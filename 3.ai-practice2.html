시작은 나만의 엑셀 전문가 챗봇부터

GPTs / GEMs가 뭔가요?
ChatGPT의 GPTs, Gemini의 GEMs는 지침(시스템 프롬프트)을 미리 넣어둔 나만의 맞춤 챗봇입니다. 한 번 잘 만들어 두면 매번 역할·조건 설명 없이 바로 전문가와 일할 수 있습니다.
"엑셀 일잘러" 챗봇의 능력
· 수식·함수 생성 + 구버전(2016/2019) 대안 동시 제공
· 스프레드시트 분석·피벗 설계·차트 추천
· VBA 자동화 코드 생성 (에러 처리 포함)
· PDF→엑셀 변환, SQL 쿼리 생성까지
// 챗봇 지침의 핵심 뼈대 (실습 1에서 전문 제공)

미션: 이 GPT는 Excel 전 과정 도우미입니다.
원칙: "최소한의 질문 → 최대한의 실행물"

응답 형식 (항상 고정):
　1. 요약 (3~5줄)
　2. 즉시 복사 블록 (수식/코드/SQL)
　3. 구현 절차 (번호 목록)
　4. 검증 / 오류 대처
　5. (옵션) 성능·가독성 개선 팁

→ 응답 형식을 지침에 박아두는 것이 핵심!
　 매번 일정한 품질의 답을 받게 됩니다
왜 챗봇부터 만드나요? 1부~2부의 모든 실습(수식, VBA, GAS)을 이 챗봇에게 시키게 됩니다. 좋은 지침을 가진 챗봇은 모든 자동화 작업의 품질을 한 단계 끌어올리는 기반 공사입니다.

메타 프롬프팅 (AI가 다듬은 것)
"이 작업을 위한 최적의 프롬프트를 먼저 만들어줘"라고 요청 → AI가 Role·Context·Task·Output 구조·Constraints를 갖춘 전문가급 프롬프트를 생성 → 그 프롬프트로 코드 생성.
메타 프롬프팅 작업 흐름
① 하고 싶은 일을 대충 설명
② "최적 프롬프트 만들어줘"
③ AI가 전문가 프롬프트 생성
④ 내가 검토·수정
⑤ 그 프롬프트로 실행

실습 5 · Julius AI로 8단계 데이터 분석
난이도 ★★☆ · 약 20분 · 3부 분석·시각화
이 매뉴얼의 목표
통계 지식이 거의 없는 초보자도 Julius AI로 ① 데이터 구조 이해 → ② 핵심 KPI 도출 → ③ 인사이트 발견 → ④ 경영 보고용 대시보드 완성까지 단계별로 따라 할 수 있도록 만든 실전 가이드입니다. 글로벌 게임 판매 데이터(제목·플랫폼·연도·장르·퍼블리셔·지역별 판매량)로 실습하지만, 내 업무 데이터로 그대로 적용해도 됩니다 (민감정보는 가명 처리!).
준비 단계
준비물: 분석할 엑셀 파일(아래 게임샘플.xlsx 또는 내 데이터) + Julius AI 계정 (무료 가입 가능)

3
엑셀 파일 업로드
분석할 파일(예: 게임샘플.xlsx)을 선택하면 업로드됩니다. 완료되면 Julius가 열(Column)을 자동으로 인식합니다.
Julius AI 접속하기 (julius.ai) 분석용 게임샘플.xlsx 다운로드
STEP 1 — 데이터 구조 이해하기 (반드시 수행)
본격적인 분석 전에 데이터가 어떤 구조인지부터 확인합니다. 이 단계를 건너뛰면 뒤 단계가 전부 흔들립니다.
열(컬럼)별로 결측값이 있는지 확인하고 요약해줘. 각 열의 데이터 타입(숫자·문자·날짜 등)을 확인해줘.
확인 포인트: 결측값이 있는가? 숫자 열이 숫자로 정확히 인식되는가? 발행연도 범위는 어떻게 되는가?
STEP 2 — 핵심 KPI 만들기
경영 보고의 기본은 KPI입니다. 아래 3가지만으로도 1차 경영 보고가 가능합니다.
1) 전체 성과 전체 글로벌 판매량(총합)을 보여줘. 2) 지역별 판매 지역별(북미, 유럽, 일본, 기타) 판매량 합계를 보여줘. 3) 연도별 성장률 글로벌 판매량의 전년 대비 성장률(YoY)을 계산해줘.
STEP 3 — 트렌드 분석
연도별 글로벌 판매량을 라인 차트로 그려줘.
해석 방법: 상승 구간은 시장 확장기, 하락 구간은 산업 전환기, 급격한 변동은 플랫폼 교체 영향 가능성으로 읽습니다.
STEP 4 — 기여도 분석 (무엇이 성과를 만들었는가)
상위 10개 게임 글로벌 판매량 기준 상위 10개 타이틀을 보여줘. 상위 퍼블리셔 글로벌 판매량 합계 기준 상위 10개 퍼블리셔를 보여줘.
인사이트 예시: 매출 집중도가 높으면 리스크 존재, 특정 퍼블리셔 의존도가 높으면 협상력 취약으로 해석합니다.
STEP 5 — 지역 믹스 분석
북미, 유럽, 일본, 기타 지역이 글로벌 판매량에서 차지하는 비중(%)을 보여줘.
해석 포인트: 북미 중심 구조인지, 일본 시장 특화 타이틀이 있는지, 유럽 시장의 성장 가능성은 어떤지 살펴봅니다.
STEP 6 — 이상징후 탐지 (전략 수립에 중요)
일본 판매량이 글로벌 판매량의 50%를 초과하는 타이틀을 찾아줘. 각 지역에서 가장 우세한(점유율이 높은) 플랫폼을 찾아줘.
STEP 7 — Executive Summary 자동 생성
성과, 주요 동인, 리스크, 다음 액션을 설명하는 경영진 요약(Executive Summary)을 5개의 불릿 포인트로 작성해줘.
이 결과는 그대로 보고서에 사용할 수 있는 수준이어야 합니다.
STEP 8 — 대시보드 구성
지금까지 만든 KPI·차트를 한 화면에 정리합니다. 권장 배치는 다음과 같습니다.
지금까지 분석한 내용을 바탕으로, 상단에는 핵심 KPI(총 글로벌 판매·지역별 판매·YoY 성장률), 중단에는 연도별 추이와 플랫폼 구성, 하단에는 Top 10 게임·Top 퍼블리셔·지역 집중도 분석을 배치한 대시보드 화면을 구성해줘.

실제 서비스 코드(backend/main.py) 기준 · 4부 RAG 챗봇 · 심화 자료
상황
회계팀이 전표를 입력하면 계정코드·부서·차대변을 잡아주는 사내 챗봇입니다. AI는 우리 회사의 계정코드 583개·부서 82개·과거 전표 1,643건을 학습한 적이 없습니다. 이걸 매번 통째로 프롬프트에 넣으면 토큰이 폭발하고 엉뚱한 항목에 휩쓸려 오답을 냅니다. 그래서 질문과 관련된 항목만 그때그때 검색해 끼워 넣는 RAG를 씁니다 — 단, RAG 혼자가 아니라 3중 방어 중 한 축입니다.
전체 구조 — RAG는 3단계 중 하나일 뿐
0. 백엔드 직답 (조건부)
1. 동적 힌트 (규칙엔진)
2. RAG 검색
3. AI 호출
4. 사후 검증
"RAG"라는 이름이 붙어 있지만 정답률을 책임지는 건 검색 하나가 아니라 RAG + 규칙 기반 힌트 + 사후 검증의 3중 구조입니다. RAG는 이 중 "관련 자료를 찾아 참고하게 해주는" 2단계만 담당합니다.
실제 예제로 따라가기 — "생산1본부 활성탄 교체, 법인카드로 15만원"
1
동적 힌트 (규칙 기반, AI 호출 전 확정) — [시스템 금액 힌트] "15만" → 150,000원 정수 확정 / [제·판 구분 힌트] 생산1본부 → 제조(43xx) 계정 사용. 금액 계산은 LLM이 아니라 Python이 직접 계산해 못 박습니다.
2
RAG 검색 (키워드 토큰 매칭) — 토큰화: content_tokens=[생산1본부·활성탄·교체], payment_tokens=[법인카드]. "활성탄" 완전 포함 매칭으로 43002501 소모품비(일반)이 최고점 매칭 (계정코드 매칭은 가중치 2점).
3
AI 호출 — system = 시스템 프롬프트 + 동적 힌트 / user = [RAG 참고자료 최대 12건] + [사용자 입력]을 합쳐 gpt-5.4-mini에 전달, 전표 표 형식으로 생성 지시.
4
사후 검증 (정규식 재검사) — 계정코드 실존 · 부서코드↔부서명 일치 · 차대변 합계 일치 . 틀리면 AI가 쓴 셀을 직접 정정하거나 ⚠️ 경고를 붙입니다.

왜 키워드 매칭인가 — 벡터 임베딩을 꺼둔 실제 이유
벡터 임베딩 검색 — 실제로 꺼져 있음
다국어 문장임베딩 모델(paraphrase-multilingual-MiniLM-L12-v2)로 코사인 유사도를 계산하는 인프라는 다 갖춰져 있습니다. 하지만 짧은 회계 문구에서는 거의 모든 후보가 0.85~0.95 구간에 뭉쳐서 나와 구분력이 없었습니다.
키워드 완전일치 — 실제 채택
token in text 방식의 완전 포함 매칭. "활성탄"이라는 단어가 있는지 없는지를 이진적으로 판단하기 때문에, 짧고 도메인이 좁은 회계 문구에서는 오히려 더 정확했습니다.

"짧은 한국어 전표 문구에서 벡터 유사도가 거의 모든 항목에 0.85~0.95로 균일하게 나와 구분력이 없고, 정확히 일치하는 항목(예: '활성탄 교체'→43002501 소모품비(일반), 유사도 0.45)이 오히려 순위에서 밀려나는 문제" — 코드 주석에 남은 실제 수정 기록

별도 벡터 DB 없이 서버가 뜰 때 이 3개 JSON을 메모리에 올려 두고, 매 요청마다 순회하며 검색합니다. 데이터가 약 2,400건으로 작기 때문에 가능한 설계입니다.
이 사례가 주는 교훈
RAG는 "벡터 검색"과 동의어가 아니다 — 검색 방식은 키워드든 벡터든 자유롭게 고를 수 있고, 도메인에 따라 단순한 쪽이 이길 수 있다.
범용 임베딩 모델이 항상 정답은 아니다 — 문장이 짧고 도메인이 좁을수록 유사도 점수가 뭉쳐 구분력이 떨어질 수 있다.
LLM에게 계산을 시키지 마라 — 금액 환산·한도 비교 같은 확정 연산은 백엔드가 직접 계산해 값으로 박아 넣는 게 안전하다.
생성 이후 검증 레이어가 필요하다 — RAG와 프롬프트가 아무리 좋아도 LLM이 지시를 어길 가능성은 남으므로, 규칙 기반 사후 검증이 마지막 안전망이 된다.
